# Threat Model (MVP) — Рубеж ИИ

Модель угроз уровня MVP. Уточняется по мере реализации. Метод — STRIDE по
потокам данных и доверительным границам.

## 1. Защищаемые активы

| # | Актив | Где хранится | Класс |
|---|-------|--------------|-------|
| A1 | Персональные данные (ПДн) во вводе/документах | временно в памяти, masked в БД | Критичный |
| A2 | Секреты (API keys, JWT, пароли, DSN) | детектируются, **не хранятся raw** | Критичный |
| A3 | Коммерческая тайна (суммы, поставщики, тендеры) | masked в БД | Высокий |
| A4 | Mapping псевдоним → raw-значение | `pseudonym_mappings`, **зашифровано** | Критичный |
| A5 | Audit trail | `audit_events`, **append-only** | Высокий |
| A6 | Политики и их версии | `policies`, `policy_versions` | Высокий |
| A7 | Учётные данные пользователей и роли | `users`, `roles` | Высокий |
| A8 | Конфигурация провайдеров LLM (endpoint, ключи) | `model_providers`, env | Высокий |

## 2. Доверительные границы

- **TB1:** браузер ↔ `rubezh-api` (внешний периметр, аутентификация).
- **TB2:** `rubezh-api` ↔ внутренние сервисы (`sanitizer`, `worker`, PostgreSQL, MinIO).
- **TB3:** `rubezh-api` ↔ внешние/облачные LLM (**наименее доверенная** граница).

## 3. Угрозы и митигации

| ID | Угроза (STRIDE) | Описание | Митигация |
|----|-----------------|----------|-----------|
| T1 | Information Disclosure | Утечка raw ПДн/секретов во внешнюю LLM | Внешние модели по умолчанию получают **только masked text**; policy engine блокирует raw corporate data в `external`/`russian_cloud` |
| T2 | Information Disclosure | Секреты попадают в application logs | Запрет логирования raw secrets; логируются только риск-классы и masked representation; тест на утечку логов |
| T3 | Tampering | Подмена/удаление записей аудита | `audit_events` append-only: триггер БД запрещает UPDATE/DELETE |
| T4 | Information Disclosure | Компрометация mapping'а псевдонимов | `pseudonym_mappings` — отдельная таблица, шифрование (AES-GCM), доступ по роли |
| T5 | Elevation of Privilege | Пользователь обходит политику или роль | Решение — только в policy engine; auth-middleware проверяет роль на каждом запросе |
| T6 | Tampering | Prompt injection в пользовательском вводе/документе | Базовая эвристическая проверき injection-маркеров (ввод и ответ); ограничение зафиксировано в §5 |
| T7 | Spoofing | Подделка запроса к внутреннему сервису | Внутренние сервисы — в изолированной docker-сети; внешний доступ только через `rubezh-api` |
| T8 | Repudiation | Отрицание пользователем своих действий | Каждое решение policy engine и каждый запрос журналируются с user_id |
| T9 | Denial of Service | Перегрузка большими документами/запросами | Лимиты размера загрузки и длины запроса; лимиты на уровне провайдеров моделей |
| T10 | Information Disclosure | Утечка чужих документов через RAG | RAG-поиск фильтруется по ACL пользователя |

## 4. Принципы безопасности (инварианты)

1. Raw secrets **никогда** не попадают в обычные application logs.
2. Audit log хранит риск-классы и **masked** representation, не raw.
3. Raw mapping хранится **отдельно** и шифруется.
4. Внешние LLM по умолчанию получают **только masked text**.
5. **Все** решения policy engine логируются.
6. Внутренние сервисы недоступны снаружи периметра.

## 5. Известные ограничения MVP

- Проверка **prompt injection** — только базовые эвристики; не защищает от
  продвинутых атак. Усиление — после MVP.
- **NER** в MVP — mock-реализация; реальная русскоязычная модель подключается
  через интерфейс без изменения архитектуры.
- **Auth** в MVP — dev-токен с claim роли; интеграция Keycloak/OIDC — после MVP.
- Шифрование mapping'ов — на уровне приложения (AES-GCM); HSM/KMS — после MVP.

## 6. Остаточные риски Итерации 8 (чат-оркестрация)

- **Межпользовательская изоляция чат-сессий отсутствует в MVP.** Dev-токен
  несёт только роль; `chat_sessions.user_id` резолвится через общий
  dev-пользователя на роль (миграция `000007`). Проверка владения сессией
  в обработчике технически выполняется, но семантически тривиальна —
  все носители одной роли делят `users.id`. Настоящая межпользовательская
  изоляция появляется с приходом OIDC (см. `docs/design/identity.md`).
- **`chat_messages` хранит только прошедшее обезличивание** (`sanitized_text`
  для запросов, ответ LLM до restore — для ответов; исходный текст и
  восстановленный ответ нигде не персистятся). **Не** «ПДн в покое
  исключены по построению» — остаются возможны при ложно-отрицательных
  срабатываниях детектора sanitizer (FN). Это остаточный риск, ограниченный
  точностью sanitizer, а не архитектурой хранения.
- **Проверка утечки в ответе LLM детерминированная** (подстрочный поиск
  raw-значений из карты псевдонимов, регистронезависимый + NFC).
  Не ловит: транслитерацию, перевод, частичное цитирование, пересказ ПДн
  через перифраз. Семантическую проверку закрывает LLM-review-фильтр
  sanitizer и/или будущая итерация.
- **Флаг `response_leak_detected`** в `audit_events.detail` означает
  «внешняя модель воспроизвела замаскированное значение» (компрометация
  маскирования провайдером), а **не** «утечка третьему лицу» — получатель
  ответа тот же субъект данных. Сигнал используется для оценки модели
  и расследования (Итерация 9).
- **SSE — псевдо-стриминг.** `Provider.Complete` блокирующий; SSE
  нарезает уже готовый и полностью проверенный ответ на чанки. Истинный
  токен-стриминг (`Provider.Stream`) — после MVP, не меняет границ
  безопасности (проверка утечки выполняется до первого `delta`).

## 6.bis. Остаточные риски Итерации 11 (RAG)

- **Mock-embeddings** не имеют семантического смысла —
  релевантность результатов RAG в MVP **случайна**. Pipeline валидируется
  end-to-end (sanitize → embed → vector-search → ACL), но качество
  возвращаемых чанков не отражает реальный поиск. **Точка расширения** —
  замена `MockEmbedder` через интерфейс `Embedder` (worker — Python
  `app/embeddings/interface.py`; api — Go `internal/llm/embedder.go`) на
  реальный векторизатор (sentence-transformers, OpenAI-совместимый или
  локальный через `trusted_local`). Замена не требует миграции схемы
  (фикс `embeddings.vector(1024)`).
- **Query plaintext не персистируется**: `audit.search_performed.detail`
  хранит только SHA-256(query)[:16] хеш для трейса. Это намеренно —
  даже компрометация audit-trail не раскроет запросов пользователей.
- **ACL-фильтрация на SQL-уровне** через `acl @>jsonb_build_array(...)` —
  выполняется в одном запросе с vector ORDER BY (план запроса
  использует index-cond на documents). Утечка чужих чанков невозможна
  через application-bug — фильтр в самом SQL.

## 7. Остаточные риски Итерации 9 (audit/incidents/шифрованные mappings)

- **Расшифровка mapping (forensics-reveal) в MVP не реализована.**
  `pseudonym_mappings.raw_value_encrypted` пишется AES-GCM, но read-side
  с reveal-flow (отдельный эндпойнт + approval-цепочка
  security_officer'a + audit `mapping_revealed`) — пост-MVP. До этого
  момента mapping существует «для будущего расследования», но активно
  не используется.
- **Ротация `MAPPING_ENCRYPTION_KEY` не поддерживается.** Смена ключа
  делает все ранее зашифрованные mappings нечитаемыми. Процедура
  key-rotation (двойное чтение, миграция) — пост-MVP вместе с
  reveal-flow.
- **Сбой шифрования mapping'а в Tx1 → потеря пользовательского
  запроса.** При технически невозможной (после `config.Load`) ошибке
  `Cipher.Encrypt` основной поток прерывается с `chat_error`,
  запрос не записывается в `chat_messages`. Это **сознательное
  fail-closed-решение**: лучше потерять запрос, чем сохранить с
  пропавшим mapping'ом (forensics-цепочка неполная).
- **`assignee_id` семантически = роль в MVP.** `users` содержит одного
  dev-пользователя на роль (миграция 000007); поэтому «Назначить мне»
  назначает роль, не конкретного человека. Реальное назначение
  появляется с OIDC (см. §6 — межпользовательская изоляция).
- **Idempotency manual-инцидентов — упрощённая.** Проверка «дубль за
  60 секунд по `audit_event_id+reporter_id`»; полноценный
  `Idempotency-Key` (RFC) — пост-MVP. Race-условие при двух POST'ах
  в один nanosecond возможно, но безопасно: оба создадут отдельные
  записи, форензика остаётся целостной.
- **Push-уведомления о новых инцидентах отсутствуют.** UI делает
  polling каждые 10s; `severity=critical` имеет до 10 секунд задержки
  визуализации. Принятое операционное ограничение MVP.
- **`incident_notes` append-only.** Исправления невозможны — ошибочную
  заметку нужно дополнить отменяющей. Это намеренно: forensics-цепочка
  действий следователя не редактируема.
- **PATCH 412 при ровно совпадающем `updated_at` (µs-точность).**
  Конкурентные edit'ы в одну µs дадут «потерянное обновление»;
  вероятность < 10⁻⁶. Полноценный `version int` — пост-MVP.
- **Manual без `audit_event_id` в MVP не поддерживается.** Инвариант
  поддерживается в коде (`POST /api/incidents` требует поле), не в БД.
  Будущее расширение reveal-flow может ослабить требование.
- **Авто-инциденты без push-сигнала о сбое.** Если `CreateAutoIncident`
  падает после успешной Tx2, в журнал пишется `incident_create_failed`
  (отдельный event-тип), но **в UI это видно только при пересмотре
  audit-log'а**. SRE-алерт на такие события — пост-MVP.
- **AAD = SHA-256(session_id‖pseudonym)[:16].** Защищает от swap-атак
  на уровне БД даже внутри одной сессии (план §Р1, закрытие MINOR-M3
  ревью v2 архитектора). Не защищает от компрометации самого ключа:
  если `MAPPING_ENCRYPTION_KEY` утёк — все mappings расшифровываемы
  атакующим. Хранение ключа — отдельный риск, см. SECURITY_CHECKLIST.
