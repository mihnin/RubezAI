# Итерация 8 — Чат-оркестрация `/api/chat` — архитектурный план (v3)

Статус: **v3 после двух ревью архитектора** (6/10 → 8.5/10 → доработка).
Закрывает критерий 5 («можно отправить chat-запрос»).
v2 устранила BLOCKER-1/2 и MAJOR-1…4; v3 устраняет MAJOR-NEW-1/2 и
MINOR-NEW-1/2/3 второго ревью.

## 1. Цель и границы

Эндпойнт `/api/chat` реализует сквозной поток безопасной маршрутизации
запроса к ИИ:

```
запрос → sanitize → policy.Decide → [транзакция: запись запроса+аудита]
       → SSE → route в LLM → проверка утечки → восстановление псевдонимов
       → [транзакция: запись ответа+аудита] → SSE-стрим → done
```

Плюс управление сессиями: `/api/chat/sessions` (список, создание).

### В границах итерации 8

- `internal/sanitizer` — HTTP-клиент к `rubezh-sanitizer` (`POST /sanitize/preview`).
- `internal/storage/chat.go` — запись `chat_sessions`, `chat_messages`,
  `sanitization_results` (транзакционно).
- `internal/storage/audit.go` — `InsertAuditEvent` (append-only запись).
- `internal/storage/users.go` — `UserIDForRole` (резолв dev-пользователя).
- `internal/chat` — оркестратор: маппинг `SanitizeResponse → policy.Input`,
  карта псевдонимов, восстановление, проверка ответа на утечку.
- `internal/api/chat.go` — обработчики `/api/chat/sessions` и `/api/chat` (SSE).
- `migrations/000007_dev_users.*` — посев dev-пользователей (по одному на роль).
- `docs/contracts/chat.schema.json` — контракт запроса, сессии и payload
  SSE-событий.
- Обновление `docs/THREAT_MODEL.md` — остаточные риски (см. §8).
- Тесты: unit (оркестратор, клиент, псевдонимы, проверка утечки),
  интеграционные (`/api/chat` сквозной, два аудит-события), контрактные.

### Вне границ (вынесено осознанно)

- **Шифрованная персистентность `pseudonym_mappings`** → Итерация 9.
  Требует Go-слоя AES-GCM (forensics-функция). Восстановление в рамках
  запроса работает без неё. В Итерации 8 таблица `pseudonym_mappings`
  остаётся **пустой**, `MAPPING_ENCRYPTION_KEY` **не читается** `config`
  и не используется (мёртвого кода не добавлять). Записывается в техдолг.
- **Истинный токен-стриминг** (`Provider.Stream`) → после MVP. `Complete`
  блокирующий; SSE итерации 8 нарезает уже готовый и проверенный ответ на
  чанки — это **псевдо-стриминг**: между `meta` и первым `delta` клиент
  видит паузу на время вызова LLM. Контракт и фронт это учитывают.
- **Авто-инцидент при `deny`/`escalate`/утечке** → Итерация 9. Итерация 8
  пишет `audit_events` и флаг `response_leak_detected`; инцидент создаётся
  в 9 на основе этих записей.
- **Восстановление псевдонимов в истории при чтении** → после Итерации 9
  (нужен персистентный mapping). В Итерации 8 `chat_messages` хранит
  псевдонимизированный текст; история показывается с псевдонимами.

## 2. Ключевые архитектурные решения

### Р1. Идентичность пользователя — посев dev-пользователей

`chat_sessions.user_id` — `NOT NULL FK → users(id)`, dev-токен несёт только
роль. Миграция `000007` сеет по одному пользователю на каждую из 6 ролей;
`storage.UserIDForRole(ctx, role)` резолвит `users.id` по коду роли. Токен
не меняется — согласуется со статусом dev-токена («MVP, заменяется на OIDC»).

**Следствие (честно):** при общей идентичности на роль межпользовательской
изоляции сессий в MVP **нет** — все носители роли `user` делят `users.id`.
Проверка владения сессией в потоке сохраняется (заработает с OIDC), но в
MVP она тривиальна. Это фиксируется в `docs/THREAT_MODEL.md` как принятое
ограничение MVP, а не как реализованная изоляция.

### Р2. Восстановление псевдонимов — карта строится в `rubezh-api`

`/sanitize/preview` stateless, сырых значений не возвращает (только
`pseudonym`, `raw_hash`, спаны `start`/`end`). Оркестратор имеет исходный
текст и строит `map[pseudonym]rawValue`:
`rawValue = string([]rune(originalText)[entity.start:entity.end])`.

- **Инвариант происхождения спанов (контракт, закрывает MAJOR-NEW-2):**
  `start`/`end` определены по код-поинтам **того же текста**, что был в
  `SanitizeRequest.text` — byte-identical, sanitizer вход не нормализует
  (проверено: в `rubezh-sanitizer/app` нет NFC/normalize/casefold).
  Это фиксируется контрактным тестом Ф1: для текста с кириллицей и
  известной сущностью `[]rune(text)[start:end]` обязан давать ровно
  ожидаемое сырое значение. От этого инварианта зависит корректность
  restore и проверки утечки (Р4) — то есть безопасность.
- Спаны от Python — смещения по **код-поинтам**; в Go обязателен перевод
  через `[]rune`.
- **Defensive-проверки (fail-closed):** для каждой сущности
  `0 ≤ start < end ≤ len([]rune(text))`; сущности сортируются по `start`;
  спаны не должны пересекаться. Нарушение инварианта → запрос не уходит в
  LLM, пишется `chat_error`, SSE `error`. Не доверяем ответу sanitizer
  слепо. Примечание: проверка границ ловит выход за пределы, но не
  «индекс указывает не туда» — последнее закрывается контрактным тестом
  происхождения спанов выше.
- Восстановление — однопроходная замена (`strings.NewReplacer`):
  псевдонимы zero-padded фиксированной ширины (`ФИО_001`), не префиксы
  друг друга — коллизий нет.

### Р3. Отображение решения политики в действие

| Decision | В LLM отправляется | Ответ пользователю |
|---|---|---|
| `allow_raw` | исходный текст | как есть (карты псевдонимов нет) |
| `allow_masked` | `sanitized_text` | проверка утечки → восстановление псевдонимов |
| `allow_summary_only` | `sanitized_text` + system «только краткое резюме» | проверка утечки → **без восстановления** (детерминированный контроль, см. ниже) |
| `deny` | — | SSE `meta`(deny, reasons) + `done`; аудит `chat_blocked` |
| `escalate` | — | SSE `meta`(escalate, reasons) + `done`; аудит `chat_blocked` |

`Input.ModelTrust` — из `trust_level` выбранного провайдера; `Input.Risk` —
из `SanitizeResponse.risk`; `EntityTypes` — из `entities[].type`;
`UserRole` — из контекста auth; `Context` — `"chat"`.

**`allow_summary_only` — детерминированный контроль (закрывает MAJOR-3).**
Это строжайший из разрешающих путей (риск `high`). Полагаться на послушание
LLM system-инструкции нельзя. Поэтому в режиме summary-only псевдонимы
**не восстанавливаются**: пользователь получает псевдонимизированное резюме.
Даже если LLM проигнорирует инструкцию и выдаст развёрнутый ответ — сырых
ПДн в нём не появится, потому что restore не выполняется. Контроль
детерминирован (отсутствие шага restore), а не вероятностен.

### Р4. Проверка ответа на утечку — на сыром ответе LLM до restore

(Закрывает BLOCKER-2.) Объект проверки — **сырой ответ LLM до любого
восстановления**. LLM получал только `sanitized_text` (псевдонимы), сырых
значений знать не должен. Для `allow_raw` карта псевдонимов не строится —
проверка утечки **пропускается** (исходный текст и так был разрешён).
Алгоритм для `allow_masked`/`allow_summary_only`:

1. Для каждого `(pseudonym → raw)` из карты искать вхождение `raw` в сыром
   ответе LLM. Сравнение **регистронезависимое**, с Unicode-нормализацией
   NFC обеих строк.
2. Найдено вхождение `raw` → утечка: в `audit.detail` ставится
   `response_leak_detected: true` и список затронутых `entity_type`.
3. Действие зависит от режима:
   - `allow_masked`: пользователь имеет право видеть свои данные —
     restore выполняется штатно; утечка фиксируется как **аудит-сигнал**
     (Итерация 9 превращает флаг в инцидент). Дополнительной экспозиции нет
     (это данные самого пользователя).
   - `allow_summary_only`: restore не выполняется; обнаруженные сырые
     вхождения **ре-маскируются** обратно в псевдоним, чтобы ответ остался
     полностью псевдонимизированным.
4. Проверка и (для summary) ре-маскирование выполняются **полностью до
   первого `delta`** — ответ к моменту стрима уже готов и проверен.

**Остаточный риск (в `THREAT_MODEL.md`):** транслитерация, перевод на
другой язык, частичное цитирование и пересказ ПДн детерминированным поиском
не ловятся. Это известное ограничение; семантическую проверку закрывает
LLM-review-фильтр sanitizer и/или будущая итерация.

### Р5. SSE на уровне обработчика — псевдо-стриминг

`text/event-stream`, `Cache-Control: no-cache`, `http.Flusher` после каждого
события. Контекст: в `Router.Complete` передаётся **`r.Context()`** (не
`context.Background()`) — отмена клиентом прерывает HTTP-вызов к LLM.
События (имя события + JSON-`data`). После доводки по ревью этапа A (M2)
все коррелированные события несут `request_id` — это критический коррелятор
для расследования инцидентов с любого момента потока:

- `meta` — `{decision, risk, provider, reasons, request_id}` — всегда первым.
- `delta` — `{content}` — чанк ответа (только при вызове LLM; после полной
  проверки утечки).
- `done` — `{request_id}` — терминальное, успех.
- `error` — `{message, request_id}` — терминальное, при сбое.

### Р6. Порядок записи в БД и аудит — два события (закрывает BLOCKER-1)

`audit_events` — append-only (триггер `rubezh_block_mutation`), поэтому
«дописать итог» = вставить **второе событие**, а не обновить первое.

**Событие 1 — `chat_request`.** Пишется в **одной транзакции** с созданием
сессии (при отсутствии), сообщением пользователя и `sanitization_result` —
**до открытия SSE и до вызова LLM**. Гарантия: ни один обработанный запрос
не остаётся без аудит-записи, даже если всё последующее упадёт.

**Событие 2 — терминальное**, одно из:
- `chat_response` — LLM вызван; в одной транзакции с сообщением ассистента;
  пишется **до** стриминга `delta`.
- `chat_blocked` — решение `deny`/`escalate`, LLM не вызывался.
- `chat_error` — сбой sanitizer/LLM/инварианта.

Терминальное событие пишется контекстом `context.WithoutCancel(r.Context())`
+ таймаут — отключение клиента не должно срывать запись аудита.

**Коррелятор (закрывает MINOR-NEW-1; после M2 ревью этапа A — расширено).**
В начале обработки генерируется `request_id` (uuid). Он кладётся в
`detail.request_id` **обоих** аудит-событий и отдаётся в SSE-событиях
**`meta`, `done` и `error`** (правка M2 после ревью этапа A — пользователь
имеет id запроса в любой момент потока, даже если до `done`/`error` поток
оборвался). `session_id` не идентифицирует пару (в сессии много запросов) —
детерминированную связку «запрос ↔ исход» в журнале даёт именно
`request_id`.

**Сбой Транзакции 2 (закрывает MINOR-NEW-3).** Если LLM успешно ответил, но
запись `chat_response` упала (сбой БД) — пишется `chat_error` с
`detail.llm_completed = true` и `detail.audit_persist_failed = true`. Это
отличает в журнале «LLM не ответил» от «ответ получен, но не сохранён» —
аудитор не должен путать эти случаи.

**Что хранит `chat_messages` (закрывает MAJOR-2; уточнено по MAJOR-NEW-1).**
`chat_messages.content` всегда хранит **псевдонимизированную форму**:
- сообщение пользователя — `sanitized_text`;
- сообщение ассистента — ответ LLM **до restore** (для summary — после
  ре-маскирования).

Исходный текст пользователя и восстановленный ответ существуют только в
памяти во время запроса, стримятся клиенту вживую и **нигде не персистятся**.

**Честная формулировка гарантии (MAJOR-NEW-1).** Хранение `sanitized_text`
вместо исходного текста снимает *основной* объём ПДн в покое, но **не даёт
абсолютной гарантии**: `sanitized_text` содержит ровно то, что распознал
sanitizer. Решение `allow_raw` срабатывает при *отсутствии распознанных*
классов `pii/secret/commercial` — это «детектор ничего не нашёл», а не «ПДн
нет». Любой ложно-отрицательный детектор (FN sanitizer) оставляет
нераспознанные ПДн и в `sanitized_text`, и (при `allow_raw`) тем самым в
`chat_messages`. Корректная формулировка для `THREAT_MODEL.md`:
«`chat_messages` хранит только прошедшее обезличивание; остаточные ПДн
возможны при пропусках детектора sanitizer — это остаточный риск, а не
исключённый по построению».

Цена решения — история показывает псевдонимы до тех пор, пока Итерация 9 не
добавит restore истории по персистентному mapping. **System-промпт**
summary-only (`«только краткое резюме»`) живёт только в `ChatRequest.Messages`
в памяти и **не персистится** в `chat_messages` (роль `system` схемой
разрешена, но не используется) — реконструкция диалога из истории не
воспроизведёт это ограничение (MINOR-NEW-2).

**Поля `audit_events` (явно, для детерминизма тестов):**
`event_type` ∈ {`chat_request`,`chat_response`,`chat_blocked`,`chat_error`};
`user_id` — резолв `UserIDForRole`; `model_provider_id` — id выбранного
провайдера (NULL для `chat_error` до выбора провайдера);
`risk_level`/`risk_classes` — из sanitize (NULL для ранних `chat_error`);
`policy_decision` — `decision` (NULL для `chat_error` до policy);
`policy_version_id` — **всегда NULL** в Итерации 8 (`DefaultPolicy`
встроенная, `policy_versions` не задействован);
`matched_rule` — `Outcome.MatchedRule`;
`masked_payload` — `sanitized_text`;
`detail` jsonb — `{session_id, entity_count, response_leak_detected?, ...}`.

**Поля `chat_messages`:** `model_provider_id` — NULL для сообщения
пользователя, id провайдера для сообщения ассистента; `role` ∈
{`user`,`assistant`}.

## 3. Поток выполнения `POST /api/chat`

0. Сгенерировать `request_id` (uuid) — коррелятор аудит-событий и поле `done`.
1. Auth → роль из контекста; `UserIDForRole(role)` → `userID`.
2. Декод тела (`decodeJSON`, строго): `{session_id?, message, provider, model?}`.
3. Валидация: `message` непустое; провайдер указан, присутствует в
   `model_providers` с `is_enabled=true` **и** зарегистрирован в `llm.Router`
   (`Router.Has`). Рассинхрон БД/Router → fail-closed: `chat_error` + SSE
   `error`. Источник `ModelTrust` — запись `model_providers`.
4. `session_id` отсутствует → создать сессию; присутствует → проверить
   принадлежность `userID` (см. Р1), иначе 404.
5. Sanitize: `sanitizerClient.Preview({text: message, context: "chat"})`.
   Сбой → `chat_error` + SSE `error`, в LLM ничего не уходит (fail-closed).
6. Построить `policy.Input`; `policy.DefaultPolicy().Decide(input)`.
   Построить карту псевдонимов (Р2) с defensive-проверками; нарушение
   инварианта спанов → `chat_error` + SSE `error`.
7. **Транзакция 1:** создать (при необходимости) сессию + сообщение
   пользователя (`sanitized_text`) + `sanitization_result` + `audit_event`
   `chat_request`. Коммит.
8. Открыть SSE; отправить `meta`.
9. Ветвление по решению:
   - `deny`/`escalate`: `audit_event` `chat_blocked`; `done`. LLM не вызывается.
   - `allow_*`: `llm.Router.Complete(r.Context(), provider, {Model, Messages})`.
     Пустой `Content` или ошибка → `chat_error` + SSE `error`.
10. Проверка утечки на сыром ответе (Р4); restore либо ре-маскирование по
    режиму (Р3/Р4) — **полностью до первого `delta`**.
11. **Транзакция 2:** сообщение ассистента (псевдонимизированная форма) +
    `audit_event` `chat_response`. Коммит.
12. Стримить `delta`-чанки (восстановленного — для masked, или
    псевдонимизированного — для summary — текста). Отправить `done`.

Любая ошибка → терминальный `chat_error` (контекст без отмены) + SSE `error`;
fail-closed: при сбое policy/sanitizer/инварианта во внешний LLM не уходит
ничего. Частный случай — сбой Транзакции 2 (шаг 11) при уже полученном
ответе LLM: клиент получает `meta`, затем `error`; `delta`/`done` **не**
отправляются (ответ не персистирован — не стримим), аудит — `chat_error`
с `detail.llm_completed=true` (см. Р6).

## 4. Файлы, пакеты и лимиты

Все файлы ≤500 строк, функции ≤60. Бизнес-логика — вне HTTP-слоя
(тестируется без сервера):

- `internal/sanitizer/` — `client.go`, `types.go`.
- `internal/storage/` — `chat.go`, `audit.go`, `users.go`.
- `internal/chat/` — `orchestrator.go` (поток), `mapping.go`
  (`SanitizeResponse → policy.Input`), `pseudonym.go` (карта + restore).
  Если `pseudonym.go` превысит лимит — дробление на `pseudonym.go`
  (карта + restore) и `leak_check.go` (проверка утечки, нормализация).
- `internal/api/chat.go` — только HTTP/SSE/декод; бизнес-логику не держит.

## 5. Контракт `chat.schema.json`

`$defs`: `ChatRequest`, `ChatSession`, `ChatSessionList`, и payload
SSE-событий — `SseMeta`, `SseDelta`, `SseDone`, `SseError`. Все объекты —
`additionalProperties: false`. Уточнение: JSON Schema описывает **payload
(`data:`)** каждого SSE-события; имена событий (`meta`/`delta`/`done`/
`error`) фиксируются текстом контракта отдельно. Контрактный тест проверяет
соответствие payload схеме. Декодирование запроса в Go — строгое
(`decodeJSON`).

`ChatRequest.message` — `minLength: 1`, `maxLength: 16384` (верхняя граница
обязательна: без неё неконтролируемо большой ввод уходит и в sanitizer, и в
LLM). Превышение → 400 до любых внешних вызовов.

## 6. План по фазам (TDD: тест-коммит → реализация-коммит)

- **Ф1.** `internal/sanitizer` — клиент + типы + контрактный тест по
  `sanitize.schema.json`, включая **тест инварианта происхождения спанов**
  (MAJOR-NEW-2): для кириллического текста с известной сущностью
  `[]rune(request.text)[start:end]` обязан давать ровно ожидаемое сырое
  значение.
- **Ф2.** `storage`: `users.go`, `chat.go`, `audit.go`, миграция
  `000007_dev_users`; тесты против реальной PostgreSQL, проверка
  append-only `audit_events`.

## 7. Согласование с существующими типами

- `policy.Input`/`Outcome`, `Decision` — используются как есть, без изменений.
- `llm.Router.Complete(ctx, name, ChatRequest)`/`ChatResponse` — как есть.
- `storage.Storage.Pool()` — для транзакций (`pgx.Tx`) в `chat.go`.
- `auth.RoleFromContext` — источник роли; новых полей в токене нет.
- `config` — добавляется чтение уже существующего `SANITIZER_URL` в клиент
  sanitizer (поле `SanitizerURL` уже есть). `MAPPING_ENCRYPTION_KEY` не
  трогается.

## 8. Изменения в `docs/THREAT_MODEL.md`

Добавляются как принятые ограничения MVP:

- Межпользовательская изоляция чат-сессий отсутствует в MVP (общая
  dev-идентичность на роль); появляется с OIDC.
- Проверка утечки в ответе LLM детерминированная; транслитерация, перевод,
  частичное цитирование и пересказ ПДн — остаточный риск.
- Флаг `response_leak_detected` означает «внешняя модель воспроизвела
  замаскированное значение» (компрометация маскирования провайдером), а не
  «утечка к третьему лицу» — получатель ответа тот же субъект.
- `chat_messages` хранит только прошедшее обезличивание; остаточные ПДн
  возможны при ложно-отрицательных срабатываниях детектора sanitizer —
  это остаточный риск, **не** исключённый по построению.

## 9. Решённые вопросы (ответы архитектора, ревью 2)

1. Р4: для `allow_masked` при утечке ответ доставляется восстановленным
   (данные самого пользователя — новой экспозиции нет), утечка идёт
   аудит-сигналом в Итерацию 9. **Принято** для MVP.
2. Р6: два аудит-события (`chat_request` + терминальное) — **принято**;
   это точнее одного события для append-only журнала (два разных факта).
3. Р3: для `allow_summary_only` отказ от restore — **принято**. В
   SSE-событии `meta` для summary передаётся поясняющий `reason`, чтобы
   фронт показал баннер «резюме обезличено» (псевдонимы — не баг).
