# Экран: Chat (`/chat`, `/chat/:sessionId`)

UX-spec ключевого экрана. Покрывает SSE-стрим, policy-decision, preview
обезличивания, summary-only, deny/escalate, leak warning. Опирается на
`docs/contracts/chat.schema.json` и `docs/design/iteration-8-chat.md`.

## Структура

### Sidebar (256px)

- Логотип + название.
- Главная навигация (Chat, Documents, …) — только разделы доступные роли.
- Разделитель.
- Кнопка `+ Новая сессия` (primary-ghost, width 100%).
- Список сессий — группировка «Сегодня / Вчера / На прошлой неделе»,
  каждая сессия — `text-sm`, текст `text-secondary`, hover bg-elev-1,
  active — `border-left: 2px solid accent` + bg-elev-1.

### Шапка

- Breadcrumbs (опц.), Cmd+K (палитра, поздняя итерация),
  переключатель темы, user-меню.

### Область сообщений

- Виртуализованный список (Tanstack Virtual). User-сообщения — справа,
  ассистент — слева; визуально различимо bg (user — `bg-elev-2`,
  assistant — `bg-elev-1` с `border-left 2px accent` слабее, opacity 0.4).
- Каждое сообщение — карточка radius-md, padding 16, max-width 720.
- Header сообщения: «Вы · 14:32» / «Ассистент · 14:32  gpt-4o-mini».

### Preview обезличивания внутри user-сообщения

- Под текстом — секция `─── обезличивание ─────`.
- Список chips с псевдонимами (`ФИО_001`, `ДОГОВОР_014`).
- На hover — tooltip с **первыми 8 символами `raw_hash`** (например
  `a4f37b9d…`) + полный SHA-256 при удержании курсора (delay 800ms) +
  подпись «raw-значение зашифровано».
  - `raw_hash` — публичный SHA-256 исходного значения (см.
    `sanitize.schema.json#Entity.raw_hash`). Одностороннее
    преобразование — показывать публично безопасно. Это намеренно
    сделано в контракте, чтобы фронт мог визуально различить
    «два разных `ФИО_001` в разных сессиях».
- Источник данных:
  - **в активном запросе** (только что отправлен) — приходит из ответа
    `POST /api/chat` (внутренний sanitize вызов) либо из явного
    `POST /api/sanitize/preview` (если пользователь нажал кнопку
    «Превью» до отправки);
  - **при reload сессии** — приходит из `GET /api/chat/sessions/:id/messages`
    в поле `messages[].sanitization_summary.entities[]` (см.
    `chat.schema.json#ChatMessage`). Если `sanitization_summary = null`
    (старые сообщения до Итерации 9) — preview-секция не отрисовывается.
- Появляется **сразу после ответа sanitizer**, до `meta`-banner'а.

### Диалог «Превью обезличивания» (composer'а)

Опциональный pre-send-просмотр (кнопка «👁 Превью» в composer'е).
Вызывает `POST /api/sanitize/preview` с текущим текстом.

Состояния диалога:

| State | Описание |
|-------|----------|
| Loading | Skeleton entity-chip'ов, текст «Проверка обезличивания…» |
| Result | Список chip'ов + chip уровня риска + кнопки «Закрыть» / «Отправить как есть» |
| Empty (нет сущностей) | «В тексте не найдено чувствительных данных»; одна кнопка «Закрыть» |
| Error | Banner `danger` с `message`; кнопка «Закрыть» |

### Policy-banner (после `meta`)

- Inline над пустым ответом ассистента (пока нет `delta`):
  - chip `decision` (цвет по семантике §2.6 ui-system.md);
  - chip `risk.level`;
  - текст `gpt-4o-mini` (model + provider trust-level в tooltip);
  - кнопка-аккордеон «▾ причины» — раскрывает список `reasons[]`.
- Высота 40px, padding 12 16, radius `sm`, `bg-elev-1`,
  `border: 1px subtle`.

### Composer (input)

- Sticky bottom, width 100% области.
- Селектор провайдера (Radix Select) — top of composer, `text-sm`.
- Textarea auto-grow 1→8 lines, placeholder «Напишите запрос…».
- Счётчик `187 / 16 384` справа внизу, цвет:
  - `text-muted` для <90%;
  - `warning` для 90–98%;
  - `danger` для ≥99%.
- Кнопки: «Превью обезличивания» (icon `eye`, ghost),
  «Отправить» (primary, с suffix-icon ►).
- Shortcut: `Cmd/Ctrl+Enter` отправляет.

Hero `text-lg`, hint `text-secondary`, 3 пример-чипа для роли `user`
(подставляют текст в composer). Composer внизу как обычно.

### S2 — Loading (POST /api/chat → ждём meta)

- User-сообщение уже в списке, под ним:
  - Skeleton policy-banner (40h) с пульсацией;
  - Микро-текст `text-xs text-muted`: «Проверка политики…»;
- Composer — disabled с цветным spinner вместо «►».

### S3 — Streaming (`meta` → `delta`)

- Policy-banner показан с реальными значениями.
- Под ним — карточка assistant message, текст наращивается; в конце —
  мигающий курсор `▌` (1s steps(2)).
- Кнопка «Отправить» меняется на «⏹ Стоп» (danger-ghost) —
  abort `EventSource` + cancel `r.Context()` на сервере.

### S4 — Done

- Курсор гасится.
- Внизу карточки assistant'а появляется footer:
  - `request_id: a4f3-7b…` (mono, обрезанный, click-to-copy);
  - timestamp `14:32:41`;
  - иконка-копия всего ответа.

### S5 — Decision `allow_summary_only`

- Над ответом — `warning`-banner:
  > **Ответ обезличен.** Псевдонимы намеренно не раскрываются (режим
  > summary-only — высокий риск). [Что это значит?]
- В тексте остаются `ФИО_001`, `ДОГОВОР_014` и т.п. — это поведение
  ожидается, не баг (см. iteration-8-chat.md §Р3).

### S7 — Decision `escalate`

- Banner `warning`:
  > **Запрос отправлен на эскалацию.** Сотрудник ИБ примет решение.
  > Аудит-запись: a4f3-7b9d-… [копировать]
- Для роли `security_officer` — дополнительная ссылка «→ Открыть
  расследование» (создаёт инцидент на этой основе).

### S8 — `response_leak_detected: true`

- В footer assistant-сообщения — **жёлтый** banner-strip:
  > ⚠ Модель воспроизвела замаскированное значение в ответе. Создан
  > инцидент №INC-0042. [Открыть]
- Для роли `user` — без ссылки (нет доступа к Incidents).
- Для `security_officer` — кликабельно.

### S9 — SSE error

- Terminal red banner внутри ответа assistant'а:
  > ✕ Поток прервался. message: «sanitizer unavailable».
  > request_id: a4f3-7b9d-… [копировать]   [Повторить]
- `request_id` берётся из `data.request_id` события `error`
  (см. `chat.schema.json#SseError`; это поле обязательно в контракте
  после правки M2).

### S10 — Network drop (EventSource closed)

- Toast (top-right, sticky): «Соединение прервано. Переподключение…»
- Авто-reconnect EventSource (стандартный поведение); при успехе —
  toast `success` «Восстановлено» (auto-dismiss 2s).

## Accessibility

- `aria-live="polite"` на assistant-message текст (обновляется по `delta`).
- `aria-live="assertive"` на deny/escalate/error banner.
- Tab: sidebar → composer → отправка → message-list.
- Esc — закрывает open dialog (превью) и тулl-tips.
- Скрин-ридер прочитывает: «policy decision: allow masked, risk
  medium, provider gpt-4o-mini» в момент появления banner'а.

## Безопасность

- Сырые значения вводимых сущностей **отображаются только** в карточке
  отправленного user-сообщения **до перезагрузки страницы** (после —
  только `sanitized_text`, как хранится в БД, см. iteration-8 §Р6).
- При reload-сессии (`GET /api/chat/sessions/:id/messages`) сообщения
  возвращаются с псевдонимизированным `content` и опциональным
  `sanitization_summary` (см. `chat.schema.json#ChatMessage`). Если
  summary присутствует — рисуем chip'ы псевдонимов с tooltip-hash;
  raw нигде не доступен. Если summary `null` (для старых сообщений до
  Итерации 9 / для assistant-сообщений) — preview-секция не
  отрисовывается.
- Над списком сообщений сессии — badge `text-muted`:
  «Архив сессии: исходный текст и восстановленный ответ не
  персистируются (THREAT_MODEL §6)».
- Никакого copy-to-clipboard для raw-значений из истории; copy
  доступен только для `sanitized_text` / assistant-ответа.
- `raw_hash` — публичный, копирование разрешено (для расследований).

- Все 10 состояний покрыты.
- Безопасность Р3/Р4 итерации 8 отражена в UI.
- AI = assistant, decision = policy — визуально разведены.
